§02 · ceilings logging through tracking API → three hard limits

act 1 · the problem

The MLflow tracking API can't carry per-model.

If every model becomes one MLflow run, three workspace-side limits compound. None are negotiable; all surface as RESOURCE_EXHAUSTED or socket exhaustion under sustained load. The 1.5M-runs-per-experiment naive shape simply doesn't terminate.

~8SKU/s
Per-experiment write throughput
Workspace-side rate cap on log_batch. Hard ceiling regardless of driver concurrency. Confirmed empirically across c=8, c=16, c=32 driver thread counts.
100kruns
Per-experiment run-count ceiling
Workspaces start refusing create_run past this. Per-region per-SKU fan-out hits it in any nontrivial deployment.
32sockets
urllib3 HTTP pool / worker
Threads block on socket allocation past the pool size. Bytes pile up on the driver → memory pressure → OOM. Tunable via MLFLOW_HTTP_POOL_* but bounded by workspace limits.
Empirical: per-model-as-run at 5 regions × 100k SKUs × 3 model_names takes ~17 hours at full driver concurrency — and that's before the per-experiment run-count ceiling kicks the job over. We need a different shape.

§04 · bundle write bucketed · sorted · memory-bounded → scales without OOM

act 2 · the approach

Three knobs. One structural rule.

The per-region parquet bundle is the serving endpoint's data store. The write must be memory-bounded (Python workers OOM at 4MB blobs × N rows) AND produce a row-group-skippable layout (sorted SKU ranges per file). Three knobs in code; one rule about how they fit together.

64
bucket_count
read-side dir fan-out · 64 was the sweet spot from iter sweep — more dirs hurt load_context (PyArrow enumeration cost); fewer hurt row-group skipping at scale
2GB
target_bytes_per_task
per-Spark-task memory bound · n_partitions = max(bucket_count, total_bytes / target) · keeps Python workers safe regardless of N_SKU
16MB
parquet.block.size
row group size · Spark default 128MB → 16MB · cold lookup reads ONE row group, so smaller = less IO per query · ~4 model blobs per group
plan_df
  .withColumn("bucket", expr("cast(substring(sku,5,10) as int) % 64"))
  .repartitionByRange(n_partitions, "bucket", "sku")    # distribute by (bucket, sku) range
  .sortWithinPartitions("bucket", "sku")                # local sort → ordered row groups (do not skip!)
  .write
    .partitionBy("bucket")                              # bucket=N/ Hive-style dirs
    .option("parquet.block.size", 16 * 1024 * 1024)
    .parquet(bundle_uri)
Critical pairing: repartitionByRange distributes by range ACROSS partitions but does NOT sort within. Skipping the explicit sortWithinPartitions takes cold p50 from 784 ms to 12,140 ms at 10k — row groups become unsorted and PyArrow has to scan the whole file.

§05 · bundle read pyarrow · lru_cache · row-group skipping → cold p50 under 1 s

act 2 · the approach

PyArrow opens one row group.

The serving endpoint instantiates a mlflow.pyfunc.PythonModel that lazy-loads on first request per SKU. Three forces combine to keep cold lookup sub-second at 20k SKU per region: partition pruning (open only the matching bucket dir), row-group skipping (open only the matching row group within a file), and LRU caching (skip everything for repeat SKUs).

ships in package · default

PyArrowLruModel

production-ready · zero startup cost
def load_context(self, context):
  self._dataset = ds.dataset(bundle_uri,
    format="parquet", partitioning="hive")
  self._get_model = lru_cache(maxsize=512)(
    self._load_model_uncached)

def _load_model_uncached(self, sku):
  bucket = _sku_bucket(sku, BUCKET_COUNT)
  table = self._dataset.to_table(
    filter=(field("bucket") == bucket) &
           (field("sku") == sku),
    columns=["model_blob_bytes"],
  )
  return pickle.loads(
    table.column("model_blob_bytes")[0].as_py())

act 3 · recommendations

Pick the knob for your constraint.

The defaults (bucket_count=64, target_bytes_per_task=2GB, parquet.block.size=16MB) are tuned for our 6-worker m5d.2xlarge cluster at 10-20k SKU/region. Four common deviations + the knob to turn for each.

if your cluster has
Smaller worker memory
turn the knob
target_bytes_per_task ↓ to 500 MB
More Spark partitions per region · ~4 files per bucket · cold lookup ~700-900 ms · safer for memory-tight workers. Tradeoff: per-cold-query opens more file footers.
if your access pattern is
Skewed (few hot SKUs)
turn the knob
cache_size ↑ to 4096
LRU absorbs the hot set entirely · cold p50 only matters for the long-tail · stays on default PyArrowLruModel. Memory cost: cache_size × per-model-size per replica.
if your access pattern is
Uniform (all cold every time)
turn the knob
switch to PyArrowFooterIndexModel
4× faster cold · pre-loaded footer index eliminates per-lookup file enumeration · pays the cost as one-time startup. Replicas need to be long-running to amortize.

Many short-lived replicas

§08 · futures ordered by complexity → pick by need
slide 09 / 10
act 3 · recommendations

Four optimizations. Ordered by lift.

The architecture today is validated to 300k models with sub-second cold lookup. The path to 1M+ SKU/region runs through these four optimizations, ordered low-effort to high. Pick by your bottleneck — none are required if you're under ~50k SKU/region.

small lift · in-config

Parallelize footer reads at load_context

startup: 30 s → ~5 s · cold p50 unchanged
FootIndex's per-replica startup time scales linearly with file count today (sequential ParquetFile opens). Swap the inner loop for a ThreadPoolExecutor across files — IO-bound work parallelizes 6-8× on the serving driver. No API surface change.
medium lift · serving-side

Vectorize predict() across SKUs in a batch

batch throughput ↑↑ · per-row predict cost amortizes
Today's predict() loops over input rows, calling model.predict([row]) per row. Group by sku at predict-time and call model.predict(group) once per distinct sku — sklearn's vectorized path saves the per-row overhead. Drop-in in PyArrowLruModel.predict.
larger lift · storage

Move model blobs out of inline parquet

unlocks 1M+ SKU/region · removes write OOM ceiling
At ~100k SKU/region the 4.4 MB inline blob becomes the bottleneck (writes get huge, even with bounded tasks). Store models as separate small files keyed by SKU — parquet holds just the lookup keys + filenames. Serving opens one ~5 MB file per cold lookup. Same row-group-skip logic on the metadata side; blob IO becomes O(1).
strategic · platform path

Lakebase point-lookup tier

sub-10 ms cold · SLA-backed · scales beyond UC Volume FUSE
When serving traffic exceeds what UC Volume FUSE handles under concurrency, sync the artifact table to a Lakebase (managed Postgres) instance via UC. LakebaseLookupModel ships as code-only in the package today — instantiate with a Lakebase instance name and you've got SELECT model_blob_bytes WHERE sku=? point lookups in sub-10 ms. The cleanest production-scale answer when latency under load matters more than zero new infra.
