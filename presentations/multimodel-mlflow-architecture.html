§01 · problem where naive logging breaks → workspace rate ceilings
slide 02 / 12
workspace rate ceilings

Three ceilings make "log everything" impossible.

At 1.5M model entries with one MLflow run per model, you'd need ~12 hours just for artifact uploads. The fix isn't more concurrency — it's reshaping the work to stay under each ceiling. The Albertsons MLflow-at-scale guidance pinpoints the failure modes; we apply every one of them.

35RPS
Artifact upload ceiling
Per workspace. Routes through mlflowdbfs by default — which silently 401s under thread-pool concurrency. Set DISABLE_MLFLOWDBFS=true before any mlflow import.
120RPS
Tracking API ceiling
create_run, end_run, log_batch — all share this budget. Async logging + log_batch(params, metrics) hide latency; they don't raise the cap.
500Kruns
Per-experiment run cap
Hard limit. With 1 experiment per region and 100k SKUs × 3 models, naive nesting hits this. We split by region and collapse the model leaf to stay under.

hexagonal architecture

§03 · pipeline generate · train · simulate · persist · log → flow + data shapes

five-stage pipeline

Generate, train once, simulate, persist, log.

Stage 2 trains one real reference RandomForest on a single (region, sku) slice; Stage 3 broadcasts the bytes; Stage 4's pandas_udf exposes a model_training_fn seam where real distributed training would slot in. The benchmark is Stage 5 — driver-side multithreaded MLflow logging.

STAGE 1

Generate mock demand

SparkDataGenerator
Cross-product of (regions × skus × rows_per_sku) emitted via mapInPandas. Schema derived from DemandRecord.
STAGE 2

Train ONE reference

ReferenceModelTrainer
RandomForestRegressor on one (region, sku) slice. Pickled to ${volume}/reference_model/model.pkl — ~5MB target.
STAGE 3+4

Broadcast + simulate

TrainingSimulator
Broadcasts bytes; pandas_udf's model_training_fn seam emits one TrainedModelRecord per row.
STAGE 4b

Persist run plan

DeltaRunPlanRepository
Writes Delta table partitioned by region. Streamed back via toLocalIterator — driver never holds it all.
STAGE 5

Log to MLflow

*LoggingStrategy
Driver-side ThreadPoolExecutor. One experiment per region. This is the benchmark.

§06 · bundle write bucketed · sorted · memory-bounded → scales to 100k+ per region
slide 07 / 12
bundle write

Three knobs. One rule.

Per-region parquet bundles backed by Unity Catalog Volumes. The structural rule: multiple files per bucket dir with non-overlapping sku ranges — so PyArrow row-group skipping works file-by-file at read time. Three knobs control where the design sits between write memory, read latency, and dir count.

64
bucket_count
read-side dir fan-out · 64 was the sweet spot from iter sweep — more dirs hurt load_context (PyArrow enumeration); fewer dirs hurt row-group skipping at scale
2GB
target_bytes_per_task
per-Spark-task memory bound · n_partitions = max(bucket_count, total_bytes / target) · keeps Python workers safe regardless of N_SKU
16MB
parquet.block.size
row group size · Spark default 128MB → 16MB · cold lookup reads ONE row group, so smaller = less IO per query · ~4 model blobs per group
plan_df
  .withColumn("bucket", expr("cast(substring(sku,5,10) as int) % 64"))
  .repartitionByRange(n_partitions, "bucket", "sku")    # range-partition by (bucket, sku)
  .sortWithinPartitions("bucket", "sku")                # local sort → ordered row groups (critical!)
  .write
    .partitionBy("bucket")                              # Hive-style bucket=N dirs
    .option("parquet.block.size", 16 * 1024 * 1024)
    .parquet(bundle_uri)
Critical pairing: repartitionByRange alone doesn't sort WITHIN partitions. Skipping the explicit sortWithinPartitions takes cold p50 from 784 ms to 12,140 ms at 10k — row groups become unsorted and PyArrow has to scan the whole file.

bundle read

Two patterns. Pick by startup cost.

Both are mlflow.pyfunc.PythonModel classes — drop-in for Databricks Model Serving. Both use lru_cache for warm hits. The split is how cold lookup is resolved: filter pushdown (zero startup cost, scanning per-query) vs pre-loaded footer index (one-time startup, near-instant per-query).

PyArrow opens the bucket dir on each query, reads parquet footers, applies row-group skipping via min/max stats. Pays a small per-file cost; zero startup cost.

at load_context:
  for each parquet file:

§05 · config sizing + tuning → two domain entities
slide 10 / 12
configuration story

Sizing vs tuning. Two entities, three drivers.

Scope (how big?) and tuning (how hard do we push the workspace?) are separate concerns. Same values reachable from CLI args, notebook widgets, and databricks.yml bundle variables — pick the entry point that fits your workflow.

GenerationSpec

sizing · scope of the demo
spec = GenerationSpec.of(
    n_regions   = 5,         # 1..10 canonical
    n_skus      = 100_000,   # per region
    rows_per_sku= 100,       # demand observations
)
# → 50M demand rows, 1.5M run-plan entries
Dials the workload. Try 5 × 100k × 100 for the full benchmark, or 2 × 100 × 10 for a smoke test in seconds.

LoggingConfig

tuning · how hard we push
LoggingConfig(
    concurrency     = 16,         # thread pool
    async_logging   = True,
    http_retries    = 7,
    http_timeout    = 120,
    run_plan_table  = "users.akil_thomas.run_plan",
    artifact_volume = "/Volumes/users/.../mlflow_artifacts",
)
Tunes the bandwidth. Higher concurrency reaches the 120 RPS tracking ceiling faster; retries + timeout absorb workspace HTTP jitter.

§06 · deploy databricks asset bundle → three resources
slide 11 / 12
deployment · databricks asset bundle

Three resources, one command.

The bundle ships a job, an all-purpose cluster, and the wheel + workspace files in one deploy. uv-managed dependencies are exported to requirements.txt and installed by the notebook via %pip install -r ../requirements.txt. Per-region experiments are created at runtime by the strategy adapters.

Jobresources/jobs.yml
spark_python_task running src/main.py on its own job cluster. Reads parameters from databricks.yml variables (strategy, regions, skus, concurrency, table names). Wheel installed as a task library.
task type: spark_python_task
cluster: job cluster · DBR 15.4 ML · Photon
library: dist/*.whl
Clusterresources/clusters.yml
All-purpose cluster for the notebook. Workers on m5d.2xlarge (32GB + 300GB NVMe instance-store — Spark spill needs the SSD at 50M-row scale). Driver on i3.4xlarge (16 vCPU / 122GB) for the ThreadPoolExecutor fanout.

§07 · run deploy · run · iterate → commands + next steps
slide 12 / 12
run it yourself

From git clone to first benchmark.

The demo is idempotent — schema, volume, and tables are created on first run via CREATE … IF NOT EXISTS. Drop the run-plan table and re-run to reseed the benchmark. Per-experiment runs are immutable; subsequent runs append, never overwrite.

Three commands.

# 1. Build deps lockfile (one-time) $uv export --format requirements-txt \ --no-hashes --no-emit-project \ --no-dev > requirements.txt # 2. Deploy bundle $databricks --profile e2-field-demo-west \ bundle deploy -t dev # 3. Run the job $databricks --profile e2-field-demo-west \ bundle run multimodel_registration -t dev

What to try next.

  • Run the notebook interactively — every cell is annotated; widgets let you dial scope without redeploying.
  • Swap --strategy nested_model to see the run-count vs throughput trade-off live.
  • Replace model_training_fn in TrainingSimulator with a real fit() → pickle.dumps to make it an actual distributed trainer.
  • Add a new logging strategy: implement ModelLoggingStrategyPort, register it in main.py, deploy.
